# ComfyUI 室內設計渲染技術規劃

**DesignClaw Render Agent 整合方案**
**硬體環境：** NVIDIA RTX 3090 (24GB VRAM)
**日期：** 2026-04-04

---

## 摘要

本文件規劃 DesignClaw 自動化管線中 Render Agent 的技術實作方案，以 ComfyUI 作為核心渲染引擎，整合 SDXL base model、Dual ControlNet（floor plan 空間約束）、IP-Adapter（風格遷移）和日式簡約風格 prompt engineering，在 RTX 3090 24GB VRAM 環境下實現 6 房間批量渲染（預估 3.5–5 分鐘），輸出至少 1024×1024 的寫實室內設計渲染圖。

## 1. 最適合室內設計的 Model 組合

### 1.1 Base Model 選擇：SDXL vs Flux vs SD 1.5

| 特性 | SD 1.5 | SDXL | Flux.1-dev |
|------|--------|------|------------|
| 原生解析度 | 512×512 | 1024×1024 | 1024×1024 |
| 寫實品質 | 中等 | 高 | 極高 |
| ControlNet 生態 | 成熟但過時 | 最成熟、preprocessor 最多 | 快速發展中，已有 canny/depth |
| LoRA 生態 | 最豐富但品質參差 | 豐富且高品質 | 持續增長中 |
| 24GB VRAM 表現 | 綽綽有餘 | 舒適運行，含 refiner 約 12–16GB | 需 FP8 量化，約 16–20GB |
| 推理速度 (3090) | 最快 | 中等 | 較慢（約比 SDXL 慢 30–50%） |

**推薦策略：以 SDXL 為主力，Flux 為備選。**

理由如下：SDXL 在 ControlNet 整合上最成熟，擁有大量室內設計相關的 fine-tuned checkpoint 和 LoRA，在 RTX 3090 24GB 上可以舒適地跑到 1024×1024 甚至 1536×1024 的解析度。Flux 在純粹的寫實品質上略勝一籌（尤其是光線和材質），但其 ControlNet 生態仍在追趕，且推理速度較慢，不利於批量渲染場景。

**推薦 Checkpoint（SDXL 系列）：**

- **RealVisXL V5.0** — 目前最受歡迎的寫實 SDXL checkpoint，對建築和室內場景表現優異
- **JuggernautXL V9** — 通用寫實品質極高，光線和材質細節出色
- **AnalogMadnessXL** — 如需偏攝影質感的渲染風格
- **DreamShaper XL** — 介於寫實與風格化之間的靈活選項

### 1.2 室內設計專用 LoRA

| LoRA 名稱 | 用途 | 建議權重 |
|-----------|------|----------|
| Interior Design SDXL LoRA | 通用室內設計風格強化 | 0.6–0.8 |
| Architecture Photography LoRA | 建築攝影光線與構圖 | 0.5–0.7 |
| lwpm-Coloring LoRA | 色彩分布優化（floor plan 上色） | 0.5–0.6 |
| Metallic/Material Texture LoRA | 材質紋理增強（金屬、木材、石材） | 0.3–0.5 |
| Japanese Interior LoRA（如有） | 日式室內風格強化 | 0.5–0.7 |

LoRA 可以疊加使用，但建議總權重不超過 1.5，避免風格過度偏移。

### 1.3 ControlNet 用法：平面圖作為空間約束

ControlNet 是將 floor plan 轉化為渲染圖的核心技術。推薦使用 **Dual ControlNet** 策略：

**方案 A：Floor Plan → Canny + Depth（推薦）**

1. **ControlNet-Canny**（權重 0.6–0.8）：將 floor plan 的線條結構作為邊緣約束，保留牆面、門窗的位置關係。使用 `flux-canny-controlnet_v3` 或 SDXL 對應版本。
2. **ControlNet-Depth**（權重 0.5–0.7）：提供空間深度資訊，讓 AI 理解房間的三維結構。可以從 floor plan 手動或自動生成 depth map。

**方案 B：Floor Plan → Lineart + Seg**

1. **ControlNet-Lineart**（權重 0.7–0.8）：保留原始結構，適合需要精確保留平面圖線條的場景。
2. **ControlNet-Seg**（權重 0.4–0.6）：用 segmentation map 標記不同區域（客廳、臥室、廚房等）。

**ControlNet Preprocessor 選擇：**

- **Canny Edge Detector** — 提取 floor plan 的邊緣
- **M-LSD (Line Segment Detector)** — 特別適合建築直線結構
- **Depth Anything V2** — 從透視圖生成 depth map
- **Scribble** — 如果輸入是手繪草圖

### 1.4 IP-Adapter：風格參考

IP-Adapter 可作為「1-image LoRA」，將參考圖片的風格遷移到新生成的渲染中。

**核心節點配置：**

- **IPAdapter Unified Loader** — 載入 IP-Adapter 模型
- **IPAdapter Advanced** — 設定 Weight Type 為 `Style Transfer (SDXL)`，用於風格遷移
- 如需保留構圖，可將 Weight Type 改為 `Composition (SDXL)`

**在 DesignClaw 中的用法：** 客戶提供「我喜歡這種風格」的參考圖 → IP-Adapter 提取風格特徵 → 結合 floor plan ControlNet 控制空間結構 → 生成符合客戶風格偏好的渲染圖。

### 1.5 Inpainting：局部修改

用於渲染後的細節調整，例如替換傢俱、更改牆面材質、調整窗簾樣式等。

**核心節點：**
- **Set Latent Noise Mask** — 定義 inpaint 區域
- **VAE Encode (for Inpainting)** — 編碼 masked 圖片
- **Fooocus Inpaint Model**（需安裝 ComfyUI Inpaint Nodes）— 提升 inpaint 一致性

## 2. ComfyUI API 整合

### 2.1 ComfyUI Server 架構

ComfyUI 啟動後會在 `http://127.0.0.1:8188` 開啟一個基於 aiohttp 的 HTTP server，同時提供 REST API 和 WebSocket 介面。

**核心 Endpoints：**

| Method | Endpoint | 用途 |
|--------|----------|------|
| POST | `/prompt` | 提交 workflow 執行 |
| GET | `/history` | 查詢歷史執行結果 |
| GET | `/history/{prompt_id}` | 查詢特定 prompt 的結果 |
| GET | `/view?filename=xxx` | 取回生成的圖片 |
| GET | `/queue` | 查看目前佇列狀態 |
| POST | `/queue` | 清除佇列 |
| GET | `/object_info` | 取得所有可用 node 類型 |
| GET | `/system_stats` | 系統狀態（VRAM、CPU 等） |
| WS | `/ws?clientId=xxx` | WebSocket 即時進度更新 |

### 2.2 Python 整合範例

```python
import json
import uuid
import urllib.request
import urllib.parse
import websocket  # pip install websocket-client

COMFYUI_URL = "http://127.0.0.1:8188"
CLIENT_ID = str(uuid.uuid4())

def queue_prompt(workflow: dict) -> str:
    """提交 workflow 到 ComfyUI，回傳 prompt_id"""
    payload = json.dumps({
        "prompt": workflow,
        "client_id": CLIENT_ID
    }).encode("utf-8")

    req = urllib.request.Request(
        f"{COMFYUI_URL}/prompt",
        data=payload,
        headers={"Content-Type": "application/json"}
    )
    resp = json.loads(urllib.request.urlopen(req).read())
    return resp["prompt_id"]

def get_history(prompt_id: str) -> dict:
    """取得特定 prompt 的執行結果"""
    url = f"{COMFYUI_URL}/history/{prompt_id}"
    resp = json.loads(urllib.request.urlopen(url).read())
    return resp

def get_image(filename: str, subfolder: str = "", folder_type: str = "output") -> bytes:
    """下載生成的圖片"""
    params = urllib.parse.urlencode({
        "filename": filename,
        "subfolder": subfolder,
        "type": folder_type
    })
    url = f"{COMFYUI_URL}/view?{params}"
    return urllib.request.urlopen(url).read()

def wait_for_completion(prompt_id: str) -> dict:
    """透過 WebSocket 等待 workflow 完成"""
    ws = websocket.WebSocket()
    ws.connect(f"ws://127.0.0.1:8188/ws?clientId={CLIENT_ID}")

    while True:
        msg = json.loads(ws.recv())
        if msg["type"] == "executing":
            data = msg["data"]
            if data["node"] is None and data["prompt_id"] == prompt_id:
                break  # 執行完成
    ws.close()
    return get_history(prompt_id)
```

### 2.3 Workflow JSON 動態修改

ComfyUI 的 workflow 以 JSON 格式儲存，每個 node 以數字 ID 為 key。透過 `File → Export (API Format)` 可以匯出 API 格式的 JSON。

```python
import copy

def modify_workflow(
    base_workflow: dict,
    positive_prompt: str,
    negative_prompt: str,
    controlnet_image_path: str,
    style_ref_path: str | None = None,
    seed: int = -1,
    width: int = 1024,
    height: int = 1024
) -> dict:
    """動態修改 workflow 參數"""
    wf = copy.deepcopy(base_workflow)

    # 修改正向 prompt（假設 node "6" 是 CLIPTextEncode）
    wf["6"]["inputs"]["text"] = positive_prompt

    # 修改負向 prompt（假設 node "7" 是 CLIPTextEncode）
    wf["7"]["inputs"]["text"] = negative_prompt

    # 修改 ControlNet 輸入圖片（假設 node "12" 是 LoadImage）
    wf["12"]["inputs"]["image"] = controlnet_image_path

    # 修改解析度（假設 node "5" 是 EmptyLatentImage）
    wf["5"]["inputs"]["width"] = width
    wf["5"]["inputs"]["height"] = height

    # 修改 seed
    if seed >= 0:
        wf["3"]["inputs"]["seed"] = seed
    else:
        import random
        wf["3"]["inputs"]["seed"] = random.randint(0, 2**32 - 1)

    # 修改風格參考圖（如有 IP-Adapter，假設 node "15" 是 LoadImage）
    if style_ref_path and "15" in wf:
        wf["15"]["inputs"]["image"] = style_ref_path

    return wf
```

**關鍵要點：** 匯出 API Format 的 JSON 後，每個 node ID 和 input key 都是固定的。建議在設計 workflow 時為關鍵 node 加上註解，方便程式化修改時定位。

### 3.3 批量處理：6 個房間排隊機制

```python
import asyncio
import time
from collections import deque

class RenderQueue:
    """Render Agent 的批量渲染佇列管理器"""

    def __init__(self, comfyui_url: str = "http://127.0.0.1:8188"):
        self.comfyui_url = comfyui_url
        self.queue: deque[RenderRequest] = deque()
        self.results: dict[str, RenderResult] = {}
        self.base_workflow: dict = {}  # 從 JSON 檔案載入

    def load_workflow(self, workflow_path: str):
        """載入基礎 workflow JSON"""
        import json
        with open(workflow_path) as f:
            self.base_workflow = json.load(f)

    async def render_room(self, request: RenderRequest) -> RenderResult:
        """渲染單一房間"""
        start_time = time.time()

        # 1. 根據 request 修改 workflow
        workflow = modify_workflow(
            self.base_workflow,
            positive_prompt=request.design_prompt,
            negative_prompt=request.negative_prompt,
            controlnet_image_path=str(request.floor_plan_image),
            style_ref_path=str(request.style_reference) if request.style_reference else None,
            seed=request.seed,
            width=request.width,
            height=request.height
        )

        # 2. 提交到 ComfyUI
        prompt_id = queue_prompt(workflow)

        # 3. 等待完成
        history = wait_for_completion(prompt_id)

        # 4. 取回圖片
        outputs = history[prompt_id]["outputs"]
        # 假設最後一個 node 是 SaveImage
        images = []
        for node_id, output in outputs.items():
            if "images" in output:
                images.extend(output["images"])

        # 5. 下載並存儲
        image_data = get_image(images[0]["filename"], images[0].get("subfolder", ""))
        output_path = Path(f"renders/{request.room_name}_{int(time.time())}.png")
        output_path.parent.mkdir(parents=True, exist_ok=True)
        output_path.write_bytes(image_data)

        elapsed = time.time() - start_time

        return RenderResult(
            room_name=request.room_name,
            image_path=output_path,
            image_url=f"/renders/{output_path.name}",
            prompt_id=prompt_id,
            generation_time=elapsed,
            metadata={"seed": request.seed, "steps": request.steps}
        )

    async def render_batch(self, requests: list[RenderRequest]) -> list[RenderResult]:
        """
        批量渲染多個房間。
        因為單張 GPU 同時只能處理一個 SDXL 推理，
        所以採用 Sequential 模式逐一處理。
        """
        results = []
        for i, req in enumerate(requests):
            print(f"[{i+1}/{len(requests)}] Rendering: {req.room_name}")
            result = await self.render_room(req)
            results.append(result)
            print(f"  ✓ Done in {result.generation_time:.1f}s")
        return results
```

**批量渲染估時（6 個房間，RTX 3090）：**

| 配置 | 單張耗時 | 6 張總計 |
|------|---------|---------|
| SDXL 1024×1024, 30 steps | ~25–35s | ~2.5–3.5 min |
| SDXL 1024×1024, 30 steps + ControlNet + IP-Adapter | ~35–50s | ~3.5–5 min |
| SDXL 1536×1024, 30 steps + ControlNet | ~45–65s | ~4.5–6.5 min |
| Flux.1 1024×1024, 20 steps + ControlNet | ~50–70s | ~5–7 min |

### 3.4 RTX 3090 VRAM 限制考量

**VRAM 使用估算：**

| 組件 | VRAM 佔用 |
|------|-----------|
| SDXL Base Model (FP16) | ~6.5 GB |
| SDXL Refiner (若需要) | ~6.5 GB |
| ControlNet Model ×1 | ~2.5 GB |
| ControlNet Model ×2 (dual) | ~5.0 GB |
| IP-Adapter Model | ~1.5 GB |
| LoRA (2–3 個) | ~0.3 GB |
| Latent + Sampling Buffer (1024×1024) | ~2.0 GB |
| VAE Decode | ~1.0 GB |
| **典型工作負載總計** | **~17–19 GB** |

**安全操作範圍：**

- 1024×1024：完全無壓力，可同時載入 dual ControlNet + IP-Adapter
- 1536×1024（或 1024×1536）：可行，但 Latent Buffer 增大，建議不同時使用 refiner
- 2048×1024：接近上限，建議使用 tiled VAE decode
- 2048×2048：超出安全範圍，必須分 tile 處理或使用 Hi-Res Fix 兩階段

**最佳化建議：**

1. 使用 `--highvram` 或 `--gpu-only` 啟動 ComfyUI 以減少 model offload 開銷
2. 不同時載入 base + refiner，改為序列執行
3. 啟用 `FP16 VAE` 減少 decode 記憶體
4. 使用 Tiled VAE（`ComfyUI-TiledDiffusion`）處理超大解析度
5. 如需超過 1536 解析度，先在 1024 生成再用 SDXL upscaler 放大

## 4. 推薦 ComfyUI Workflow 節點組合

### 4.1 主要 Workflow：Floor Plan → Interior Render

```
節點連接順序：

LoadCheckpoint (SDXL)
  ├─▶ CLIPTextEncode (正向 prompt) ─────────────────────┐
  ├─▶ CLIPTextEncode (負向 prompt) ─────────────────────┤
  └─▶ [MODEL output]                                    │
        │                                                │
        ├─▶ LoRALoader ×2 (Interior Design + Arch Photo) │
        │     └─▶ [MODEL output]                         │
        │           │                                    │
        ▼           ▼                                    │
LoadImage (floor plan)                                   │
  └─▶ CannyEdgePreprocessor                             │
        └─▶ ControlNetLoader (SDXL Canny)               │
              └─▶ ControlNetApplyAdvanced ◀──────────────┤
                    │                                    │
LoadImage (style ref, optional)                          │
  └─▶ IPAdapterUnifiedLoader                            │
        └─▶ IPAdapterAdvanced ◀──────────────────────────┘
              │
              ▼
EmptyLatentImage (1024×1024)
  └─▶ KSampler
        ├─ model: [from IPAdapter/ControlNet chain]
        ├─ positive: [from CLIPTextEncode]
        ├─ negative: [from CLIPTextEncode]
        ├─ latent: [from EmptyLatentImage]
        ├─ sampler: dpmpp_2m
        ├─ scheduler: karras
        ├─ steps: 30
        └─ cfg: 7.0
              │
              ▼
        VAEDecode
              │
              ▼
        SaveImage (output)
```

## 5. 日式簡約風格 Prompt Engineering

### 5.4 Prompt Engineering 進階技巧

**CFG Scale 建議：** 日式簡約風格建議使用 CFG 6.0–7.5，過高的 CFG 會讓畫面過度銳化，失去 wabi-sabi 的柔和質感。

**取樣器建議：** `DPM++ 2M Karras` 或 `Euler a`，30 步為佳。

**ControlNet 權重微調：** 日式風格的空間通常較為開放、簡潔，ControlNet 權重可設為 0.5–0.65（比一般略低），讓 AI 有更多自由度詮釋空間氛圍。

## 6. 實施路線圖

### Phase 1：基礎建置（1–2 週）

1. 安裝 ComfyUI + 所有 custom nodes
2. 下載 RealVisXL V5.0 + ControlNet models + IP-Adapter
3. 在 ComfyUI GUI 中手動建立基礎 workflow
4. 匯出 API Format JSON 作為模板
5. 驗證 RTX 3090 上的 VRAM 使用和生成品質

### Phase 2：API 整合（1–2 週）

1. 實作 Python RenderAgent 類別
2. 實作 workflow JSON 動態修改邏輯
3. 實作 WebSocket 進度監聽
4. 實作批量渲染佇列
5. 加入錯誤處理和重試機制

### Phase 3：品質調優（1–2 週）

1. 針對日式風格微調 prompt 模板
2. 測試不同 ControlNet 權重組合
3. 測試 IP-Adapter 風格遷移效果
4. 建立 A/B 測試框架評估渲染品質
5. 優化 VRAM 使用（Tiled VAE 等）

### Phase 4：生產整合（1–2 週）

1. 整合到 DesignClaw 主管線
2. 加入渲染結果快取機制
3. 實作健康檢查和監控
4. 壓力測試（連續渲染 50+ 張）
5. 文件化和部署流程
