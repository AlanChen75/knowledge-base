# Claude Code 會計三表案例分析

## 摘要

用 Claude Code 在 3 小時內從散落的 PDF、Excel、銀行對帳單和收據截圖，產出完整的會計三表（損益表、資產負債表、現金流量表），通過會計師查帳。這個案例展示了 AI 落地到專業領域的完整方法論：自建 skill 注入領域知識 → 分輪次處理不同資料源 → 互動式補齊缺失資訊 → subagent 紅隊演練品質把關。

## 背景

連鎖餐飲業加盟主隔天會計要來查帳，需要快速產出會計三表。靈感來自日本稅務師畠山謙人用 Claude Code 一個人服務 60 家顧問公司的案例。

## 分析內容

### 工作流程拆解

整個流程分為五個階段，每個階段都有明確的方法論意義：

#### 階段一：自建領域 Skill（/taiwan-tax）

**做了什麼：** 搜尋 GitHub 沒有現成台灣會計工具，於是自己寫了一個 Claude Code skill，包含：
- 台灣會計科目對應規則
- 含稅未稅轉換公式
- 401 營業稅雙月申報邏輯
- 哪些可以自動化、哪些不能

**方法論意義：** 這是整個案例最關鍵的一步。AI 不知道「外送佣金 28% 歸佣金支出、代扣繳稅款歸資產科目」這種領域判斷。Skill 等同於畠山說的「業務手冊」——把專業知識結構化後注入 AI。沒有這一步，後面全部白做。

**可複製性：** 任何專業領域要用 AI 落地，第一步都是建 skill。會計有會計的 skill，法務有法務的 skill，UX 研究也有自己的。

#### 階段二：結算表 → 損益表

**做了什麼：** 6 個月結算表 Excel 丟進去，Claude Code 逐頁提取數字，17 種代扣項目自動對應會計科目，自己做交叉驗證。

**方法論意義：** 結構化資料（Excel）的處理是 LLM 最穩定的能力區間。自動交叉驗證是關鍵——讓 AI 自己檢查自己，在產出的同時建立基本的品質保證。

#### 階段三：銀行對帳單補齊

**做了什麼：** 320 筆銀行交易 PDF 丟進去，自動分類後加進損益表。注意到 PDF 容易跑版，千位小數點需要人工檢查。

**方法論意義：** PDF 是 AI 處理最容易出錯的格式。作者明確標記了風險點（千位小數點），這就是人類在 AI 工作流中的正確角色——不是自己算數字，而是知道哪裡容易出錯。

#### 階段四：互動式補齊（6 個問題）

**做了什麼：** 資產負債表需要更多資訊，Claude Code 列出 6 個「待補」項目，逐一問老闆：資本額、設備成本、加盟合約年限、折舊方法、加盟金付款方式、是否含稅。每回答一個，報表更新一次。

**方法論意義：** 這是最高價值的環節。AI 知道自己缺什麼資訊，主動提問。這跟 Anthropic Harness 文章中 Planner 的角色一樣——把模糊的需求展開成具體的問題。「6 個問完，全部消除」代表 AI 有能力追蹤自己的知識缺口。

#### 階段五：Subagent 紅隊演練

**做了什麼：** 派一個 subagent 扮演台灣會計師，模擬「明天查帳會抓出什麼漏洞？」結果抓出 13 項重大發現，包括加盟金增資沒辦變更登記。

**方法論意義：** 這就是 Anthropic 那篇文章的核心論點——讓生成器自己評估自己會自我感覺良好（inflated self-assessment），必須拆開成獨立的 Evaluator。subagent 紅隊演練 = 獨立 Evaluator + 對抗性測試。13 項發現裡有些（如變更登記）是純法規層面的，單靠生成階段的 AI 絕對不會主動提醒。

### 角色定位分析

作者自述的角色：「很懂得如何跟 AI 溝通的 PM」。拆解來看：

| PM 能力 | 在這個案例中的體現 |
|---------|-------------------|
| 需求拆解 | 把「做會計三表」拆成 5 個階段 |
| 資源調度 | 知道什麼時候動用 skill、什麼時候用 subagent |
| 品質把關 | PDF 小數點人工檢查、紅隊演練 |
| 風險管理 | 知道 AI 幻覺的高風險區間在哪 |
| 溝通能力 | 能從老闆那裡問出正確的補充資訊 |

這跟 FDE（Forward Deployed Engineer）的定義高度吻合：能讀懂業務痛點、設計解決方案、親手落地。

| Harness 角色 | 本案例對應 |
|-------------|-----------|
| Planner | /taiwan-tax skill + 階段規劃 |
| Generator | Claude Code 逐輪處理資料 |

### 可複製的方法論

這個案例可以抽象成一套通用方法：

1. **建 Skill** — 把領域專業知識結構化注入 AI
2. **分輪次處理** — 按資料源和複雜度拆成多輪，每輪有交叉驗證
3. **互動式補齊** — 讓 AI 列出知識缺口，人類填補
4. **紅隊演練** — 用獨立的 subagent 做對抗性測試
5. **人類把關高風險環節** — PDF 解析、數字精確度、法規合規性

這套方法不限於會計，任何「把散落資料整理成結構化專業文件」的任務都適用：法務合約審查、醫療報告彙整、投標文件準備、ESG 報告撰寫。

### 成本與效率分析

| 指標 | 傳統做法 | Claude Code |
|------|---------|-------------|
| 時間 | 1-2 天（會計師工時） | 3 小時 |
| 人力 | 需要會計專業 | 需要 AI PM 能力 + 業主回答 6 個問題 |
| 品質保證 | 會計師經驗判斷 | skill 規則 + 交叉驗證 + 紅隊演練 |
| 可複製性 | 依賴個人經驗 | skill 可重複使用，流程可標準化 |

### 風險與限制

1. **專業判斷的邊界。** AI 能做分類和計算，但「加盟金增資要不要辦變更登記」這種法規判斷，是紅隊演練才抓到的。沒有專業 skill 和紅隊步驟，這類遺漏會直接影響查帳結果。
2. **PDF 解析的不確定性。** 千位小數點跑版問題在銀行對帳單中很常見，需要人工抽檢。
3. **資訊完整性依賴人類。** 6 個問題如果老闆答錯或遺漏，報表就會有偏差。AI 能問出問題，但無法驗證答案的正確性。
4. **skill 的維護。** 台灣稅法每年都有修正，/taiwan-tax skill 需要定期更新。

## 結論與建議

這個案例最有價值的不是「AI 能做會計三表」，而是展示了一套可複製的 AI 落地方法論：skill 注入 → 分輪處理 → 互動補齊 → 紅隊品管。這套方法論適用於任何需要把散落資料整理成專業文件的場景。

對 Alan 來說，這個案例直接驗證了 AI 工具實戰系列的核心主張：AI 不是取代專業，而是讓一個懂得溝通的 PM 能夠駕馭專業級的產出。
