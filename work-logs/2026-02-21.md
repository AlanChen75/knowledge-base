## [11:50] AI Hub 使用體驗回饋（Podcast 字稿批次生成）

### 發現問題

1. **gemini_chat 卡死偵測不足**
   -  顯示 BUSY，但實際 provider 完全無法回覆（No response from Gemini）
   - 建議：加入 health check timeout，如果 gemini_chat 連續 N 次 No response，自動標記為 DEGRADED 或 ERROR
   - 建議：自動重啟 gemini-chat Chrome 的機制

2. **Chrome renderer 進程爆炸**
   - gemini Chrome (port 9222) 的 renderer 進程從 limit=2 變成 10+
   - 導致 ac-mac 記憶體壓力，連帶影響 gemini-chat (9226)
   - 建議：定期檢查 renderer 進程數量，超過閾值自動 kill + restart

3. **Groq rate limit 資訊不足**
   - HTTP 429 只 log WARNING，沒有記錄 retry-after 時間
   - `check_ai_status` 不反映 rate limit 狀態，仍顯示 READY
   - 建議：429 後標記 groq_llm 為 RATE_LIMITED(N秒)，在 status API 反映

4. **LLM 兩個 provider 同時掛時的處理**
   - gemini_chat 卡死 + groq rate limit = 直接 503
   - 建議：增加排隊機制，等待 cooldown 後自動重試

### 使用場景
- 批次生成 107 集 podcast 字稿
- 先用 Groq 速度快但被 rate limit（連續呼叫間隔太短）
- 切到 Gemini Chat 發現 Chrome 已經異常
- 最終手動重啟 Chrome 後恢復

### 建議的改善優先級
1. Chrome health check + auto restart（高）
2. Provider 狀態精確反映（中）
3. Rate limit cooldown tracking（中）
4. LLM 排隊/延遲重試（低）
